PoC for NANOG Kea + Stork workshop

VMs

  • Each participant will get 4
    • "client": will be used for various perfdhcp commands to simulate DHCP traffic.
    • "kea1": will be used for kea-dhcp4, kea-dhcp6, and stork-agent
    • "kea2": will be used for kea-dhcp4, kea-dhcp6, and stork-agent
    • "stork": will be used for the stork-server GUI
  • These will be instantiated using https://www.linode.com/

  1. Participants do step 8
  2. Eddy gives stork-server and stork-agent overview about how the technology works.

  3. Darren demonstrates installing stork-server on "stork".

    • add the repo: curl -1sLf 'https://dl.cloudsmith.io/public/isc/stork/setup.deb.sh' | sudo -E bash
    • install the server: apt install isc-stork-server
    • Setup stork-server user and create database: sudo -u postgres stork-tool db-create --db-name stork --db-user stork-server
    • Change user used with the database by stork-server:
      • edit the file: nano /etc/stork/server.env and find the line # STORK_DATABASE_USER_NAME=. Change this line to: STORK_DATABASE_USER_NAME=stork-server and find the line: # STORK_REST_PORT= and change this line to: STORK_REST_PORT=4780
      • Press ctrl+o to save and ctrl+x to exit
    • Start the server: systemctl enable --now isc-stork-server
    • Check the status: systemctl status isc-stork-server
    • The stork server should be available at http://IP_address_of_your_server:4780

  4. Participants do step 9

  5. Darren demonstrates installing stork-agent on "kea1" and "kea2" and registering on "stork".
    • On both add the repo: curl -1sLf 'https://dl.cloudsmith.io/public/isc/stork/setup.deb.sh' | sudo -E bash
    • Install the agent: apt install isc-stork-agent
    • Register the agent: sudo -u stork-agent stork-agent register --server-url http://192.168.1.4:4780
    • Start the agent: systemctl enable --now isc-stork-agent
    • Login to the Stork UI: http://IP_address_of_your_server:4780 (Username: admin. Password: admin). You will be forced to change your password. Change it to the same password used to log into these systems.
    • Click the "unauthorized machines" link in the banner near the top complaing about "Unregistered machines".
    • You should see "kea1" and "kea2" listed here. Check the box for each and click "Authorize Selected"
    • Now refresh the web page.
    • Click the hamburger menu and choose "DHCP" (DHCP may not be under the hamburger menu depending on width of page). Click "Dashboard"
    • Should see "Green checkmark" in "Status" for "kea1" and "kea2".

  9. Now Darren does exercise 5
    • Connect to the client machine a second time as you will need two windows in which to run perfdhcp simultaneously (once for dhcp4 and once for dhcp6).
    • in the dhcp4 client window type: perfdhcp -l eth1 -4 -r 10 -R 100 -Y 1 -y 3600 -t 15
    • In the dhcp6 client window type: perfdhcp -l eth1 -6 -r 10 -R 100 -e address-and-prefix -Y 1 -y 3600 -t 15
    • Now shut down kea-dhcp4 on kea1: systemctl stop isc-kea-dhcp4-server
    • Should start to see drops in your dhcp4 client window
    • On kea2 execute the following: echo '{ "command": "status-get" }' | socat unix:/run/kea/socket4 - | jq a few times. Eventually you should see "server-name": "server2", with "state": "partner-down", and "last-state": "unavailable", with "server-name": "server1", in the "remote": { } section.
    • Look at "DHCP > Dashboard" in the Stork interface and there should be a warning triangle and "partner-down" in the "HA State" for kea2. kea1 should have a red X in status and "unavailable" in the HA State.
    • Drops should stop incrementing in your dhcp4 client window.
    • Now, repeat this for kea-dhcp6. Shut down kea-dhcp6: systemctl stop isc-kea-dhcp6-server on kea1.
    • Should start to see drops in your dhcp6 client window
    • on kea2 execute the following: echo '{ "command": "status-get" }' | socat unix:/run/kea/socket6 - | jq a few times. You should again see the same result of partner-down as described earlier.
    • Over in the Stork interface, click the refresh button next to "DHCP Dashboard". You should see similar states for the "DHCPv6" services as previously seen in "DHCPv4".
    • You will notice in the "Events" log on the Stork interface that times for the beginnings of the trouble are shown.
    • Drops should stop incrementing in your dhcp6 client window.
    • Now start both services back up on kea1: systemctl start isc-kea-dhcp4-server and systemctl start isc-kea-dhcp6-server
    • You can watch the state changes with the API commands shown earlier and also the "Events" log in Stork, and finally, by clicking the refresh button next to "DHCP Dashboard" in the Stork interface (note that this part of Stork does not auto-refresh).
    • Stop perfdhcp in the dhcp4 and dhcp6 client windows.

Notes

  • We will use port 7223 for ssh because some people attack port 22 all the time (causing ssh to not answer).
    • e.g., ssh -p 7223 root@104.237.139.215
  • We will set the hostnames to that in quotes in the VMs section above.
  • We need to preinstall socat and jq: apt install jq socat
  • We need to preinstall postgres and configure: sudo apt install postgresql and sudo systemctl enable --now postgresql
  • We need to setup "kea1" and "kea2" /etc/hosts to point the hostname at the private eth1 IP.
  • We need to setup "kea1" and "kea2" in /etc/hosts on "stork" also because no dns server :)
  • We have decided not to show editing host reservations with Stork because we would need to have database support in Kea which would complicate things. We could add a reservaiton in the configuration itself if someone thinks that is a good idea, but Stork will only show it, not manage it.
